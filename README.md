# 웨어러블 생체신호 위험 이벤트 분석

웨어러블 센서 데이터를 MySQL과 MongoDB에 각각 저장하고, 데이터 정제·집계 분석·위험 이벤트 탐지를 수행한 프로젝트입니다.

SQLAlchemy Core와 PyMongo를 활용하여 관계형 데이터베이스와 NoSQL 데이터베이스의 저장 및 분석 방식을 비교했습니다.

---

## 프로젝트 개요

웨어러블 기기에서는 심박수, 산소포화도, 체온, 호흡수, 걸음 수와 같은 생체신호가 지속적으로 생성됩니다.

본 프로젝트에서는 웨어러블 생체신호 데이터를 이용하여 다음 과정을 구현했습니다.

```text
CSV 데이터 불러오기
        ↓
MySQL 및 MongoDB 저장
        ↓
센서 오류와 비현실적 값 제거
        ↓
활동별 평균 생체지표 분석
        ↓
고심박·고체온·저산소 이벤트 탐지
        ↓
시간대별 위험 패턴 분석
        ↓
복합 위험 조건 탐지
        ↓
SQLAlchemy와 PyMongo 결과 비교
```

본 프로젝트의 위험 조건은 데이터베이스 및 분석 실습을 위해 설정한 기준이며, 실제 의료 진단 기준으로 사용하지 않습니다.

---

## 프로젝트 목표

- 웨어러블 센서 데이터의 구조와 품질 확인
- 동일한 데이터를 MySQL과 MongoDB에 저장
- 센서 오류 및 비현실적인 측정값 제거
- 활동별 평균 생체지표 분석
- 조건 기반 위험 이벤트 탐지
- 시간대별 생체지표와 위험 이벤트 패턴 분석
- 여러 위험 조건이 동시에 발생한 복합 위험군 탐지
- SQLAlchemy Core와 PyMongo 집계 결과 비교

## 데이터 구성

원본 데이터는 총 `5,000개`의 웨어러블 센서 기록과 `14개`의 변수로 구성되어 있습니다.

### 주요 변수

| 변수 | 설명 |
|---|---|
| `timestamp` | 측정 시간 |
| `heart_rate` | 심박수 |
| `blood_oxygen` | 산소포화도 |
| `body_temp` | 체온 |
| `resp_rate` | 호흡수 |
| `step_count` | 걸음 수 |
| `activity` | 활동 종류 |
| `accel_x`, `accel_y`, `accel_z` | 3축 가속도 센서값 |
| `gyro_x`, `gyro_y`, `gyro_z` | 3축 자이로 센서값 |
| `is_tampered` | 센서 오류 또는 변조 여부 |

### 활동 종류

- `Walking`
- `Cycling`
- `Running`
- `Resting`

### 데이터 시간 범위

```text
시작 시각: 2024-04-01 00:00:00
종료 시각: 2024-04-01 13:53:10
```

## 데이터베이스 구성

같은 웨어러블 데이터를 두 종류의 데이터베이스에 각각 저장했습니다.

| 구분 | 데이터베이스 | Python 도구 | 저장 형태 |
|---|---|---|---|
| 관계형 데이터베이스 | MySQL | SQLAlchemy | Table |
| NoSQL 데이터베이스 | MongoDB | PyMongo | Document |

### MySQL

```text
Database: wearable_db

원본 테이블:
biometric_data

정제 테이블:
biometric_data_clean
```

### MongoDB

```text
Database: wearable_db

원본 Collection:
biometric_records

정제 Collection:
biometric_records_clean
```

원본 데이터 5,000개가 MySQL과 MongoDB에 모두 정상적으로 저장되었는지 행과 Document 개수를 비교하여 확인했습니다.

> 본 기준은 데이터 분석 실습을 위해 설정한 조건이며 실제 의학적 진단 기준이 아닙니다.

## 분석 내용

### 1. 활동별 평균 생체지표 분석

정제된 4,240개의 데이터를 활동별로 묶어 다음 평균값을 계산했습니다.

- 평균 심박수
- 평균 산소포화도
- 평균 체온
- 평균 호흡수
- 평균 걸음 수

SQLAlchemy Core에서는 다음 기능을 사용했습니다.

```text
select
func.avg
func.count
group_by
order_by
```

PyMongo에서는 Aggregation Pipeline의 다음 기능을 사용했습니다.

```text
$group
$avg
$sum
$project
$sort
```

### 활동별 평균 결과

| 활동 | 데이터 수 | 평균 심박수 | 평균 산소포화도 | 평균 체온 | 평균 호흡수 | 평균 걸음 수 |
|---|---:|---:|---:|---:|---:|---:|
| Resting | 1,012 | 121.95 | 94.92 | 37.51 | 18.21 | 98.80 |
| Walking | 1,108 | 119.77 | 94.98 | 37.48 | 18.06 | 102.33 |
| Cycling | 1,063 | 119.34 | 94.89 | 37.41 | 17.91 | 98.63 |
| Running | 1,057 | 118.55 | 95.17 | 37.52 | 18.06 | 98.80 |

### 주요 결과

- 평균 심박수가 가장 높은 활동: `Resting` — 121.95
- 평균 산소포화도가 가장 높은 활동: `Running` — 95.17
- 평균 체온이 가장 높은 활동: `Running` — 37.52
- 평균 호흡수가 가장 높은 활동: `Resting` — 18.21
- 평균 걸음 수가 가장 높은 활동: `Walking` — 102.33

SQLAlchemy Core와 PyMongo로 동일한 집계 분석을 수행한 결과, 두 방식의 계산 결과가 동일하게 나타났습니다.

데이터의 출처와 생성 방식이 명확하지 않기 때문에 Resting의 평균 심박수가 가장 높다는 결과를 실제 생리적 현상으로 단정하지 않았습니다.

## 활동별 위험 이벤트 분석

고심박, 고체온, 저산소 중 하나 이상에 해당하는 데이터를 활동별로 집계했습니다.

| 활동 | 전체 데이터 | 위험 이벤트 수 | 위험 이벤트 비율 |
|---|---:|---:|---:|
| Walking | 1,108 | 786 | 70.94% |
| Resting | 1,012 | 715 | 70.65% |
| Running | 1,057 | 718 | 67.93% |
| Cycling | 1,063 | 721 | 67.83% |

위험 이벤트 비율이 가장 높은 활동은 `Walking`으로 나타났습니다.

```text
위험 이벤트 수: 786
위험 이벤트 비율: 70.94%
```

활동별 차이는 크지 않았으며, 모든 활동에서 위험 이벤트 후보 비율이 높게 나타났습니다.

따라서 특정 활동이 실제로 더 위험하다고 단정하기보다는, SQLAlchemy Core를 활용한 조건 검색 및 집계 분석 결과로 해석했습니다.

## 복합 위험 조건 분석

### 분석 조합

- 고심박 + 고체온
- 고심박 + 저산소

- 고심박 + 고체온 + 저산소

### 위험 조건 2개 이상 발생 비율

복합 위험 조건 비율이 가장 높은 활동은 `Resting`으로 나타났습니다.

### 세 가지 조건이 모두 발생한 데이터

## SQLAlchemy와 PyMongo 비교

| 도구 | 연결 데이터베이스 | 프로젝트 내 역할 |
|---|---|---|
| SQLAlchemy | MySQL | 테이블 저장, 조건 검색, 집계 분석, 위험 이벤트 탐지 |
| PyMongo | MongoDB | Document 저장, Aggregation Pipeline 기반 집계 |
| Pandas | DataFrame | 데이터 확인, 정제, 원본·정제 데이터 비교 |

### SQLAlchemy Core에서 사용한 기능

```text
select
func.avg
func.sum
func.count
case
and_
or_
where
group_by
order_by
```

### PyMongo에서 사용한 기능

```text
insert_many
delete_many
count_documents
aggregate
$group
$avg
$sum
$project
$sort
```

SQLAlchemy는 구조화된 테이블 기반 조건 분석에 사용했고, PyMongo는 Document 형태의 데이터 저장과 Aggregation Pipeline 기반 집계에 사용했습니다.

- 높은 위험 이벤트 비율은 데이터의 생성 방식이나 분포 특성에서 비롯되었을 가능성이 있습니다.

## 프로젝트 결론

본 프로젝트에서는 웨어러블 생체신호 데이터를 MySQL과 MongoDB에 각각 저장하고, 데이터 정제부터 조건 기반 위험 이벤트 탐지까지 전체 분석 흐름을 구현했습니다.

SQLAlchemy Core를 이용하여 MySQL 기반 조건 검색과 집계 분석을 수행했으며, PyMongo Aggregation Pipeline을 이용해 MongoDB 기반 집계 분석을 구현했습니다.

동일한 활동별 평균 분석 결과가 두 데이터베이스에서 일치하는 것을 확인했으며, 관계형 데이터베이스와 NoSQL 데이터베이스를 모두 활용하여 데이터를 저장하고 분석하는 과정을 경험했습니다.
